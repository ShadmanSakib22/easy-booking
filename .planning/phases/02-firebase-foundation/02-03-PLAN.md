Overwrite `firestore.rules` at the repo root with the following complete content.

CRITICAL RULES BEFORE WRITING:
1. `rules_version = '2';` MUST be the first line — required for correct wildcard behavior.
2. On `create` operations, `resource.data` is null. Use `request.resource.data` to read incoming data on creates.
3. On `update`/`delete` operations, `resource.data` is the existing document.
4. `allow read, write: if false` on invites means NO client operation succeeds — Admin SDK bypasses rules entirely by design.
5. The calendars create rule uses `request.resource.data.creatorId` (incoming data) not `resource.data.creatorId` (which would be null on create).

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // ── users/{userId} ──────────────────────────────────────────────────────
    // Authenticated user can read and write only their own document.
    // No cross-user reads (profile data is accessed via public profile endpoint).
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }

    // ── calendars/{calendarId} ──────────────────────────────────────────────
    // Public read: any visitor can read published+public calendars (booking pages).
    // Creator read+write: creator has full access to their own calendars.
    // Creator create: new calendar must have creatorId == requesting user.
    // Non-creator writes: denied — only the owning creator can modify.
    match /calendars/{calendarId} {
      // Public read — status AND visibility must both match
      allow read: if resource.data.status == 'published'
                  && resource.data.visibility == 'public';

      // Creator full access (read + update + delete)
      allow read, update, delete: if request.auth != null
                                  && request.auth.uid == resource.data.creatorId;

      // Creator create — use request.resource.data (resource.data is null on create)
      allow create: if request.auth != null
                    && request.auth.uid == request.resource.data.creatorId;
    }

    // ── slots/{slotId} ──────────────────────────────────────────────────────
    // Public read: slot availability is public information needed for booking pages.
    // Creator write: only the calendar creator can create/update/delete slots.
    // Slot status transitions (pending → booked) go through Admin SDK in booking flow.
    match /slots/{slotId} {
      allow read: if true;

      // Creator update + delete
      allow update, delete: if request.auth != null
                            && request.auth.uid == resource.data.creatorId;

      // Creator create — creatorId on the new slot must match auth uid
      allow create: if request.auth != null
                    && request.auth.uid == request.resource.data.creatorId;
    }

    // ── bookings/{bookingId} ────────────────────────────────────────────────
    // Creator read: creator can read all bookings on their calendars (DASH-01).
    // Booker read: booker can read their own booking (by bookerUid).
    // All writes: denied — booking creation and status updates go through Admin SDK.
    // This enforces the atomic transaction pattern (BOOK-05) server-side only.
    match /bookings/{bookingId} {
      allow read: if request.auth != null
                  && (request.auth.uid == resource.data.creatorId
                      || request.auth.uid == resource.data.bookerUid);

      // All writes via Admin SDK (bypasses these rules)
      allow write: if false;
    }

    // ── invites/{inviteId} ──────────────────────────────────────────────────
    // No client reads or writes — all OTP operations go through server-side
    // Route Handlers using Admin SDK (INV-03, INV-04).
    // Admin SDK bypasses security rules entirely — these rules affect client SDK only.
    match /invites/{inviteId} {
      allow read, write: if false;
    }

  }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| unauthenticated client → Firestore | Rules deny all writes for unauthenticated users |
| authenticated non-creator → creator resources | Rules deny cross-user calendar/slot writes |
| client SDK → bookings/invites | Rules deny all; Admin SDK is the only writer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-03-01 | Tampering | Unauthenticated write to users collection | mitigate | `request.auth != null && request.auth.uid == userId` — unauthenticated request has `request.auth == null` which fails |
| T-2-03-02 | Tampering | Non-creator modifying another user's calendar | mitigate | `request.auth.uid == resource.data.creatorId` check on update/delete; create rule checks `request.resource.data.creatorId` |
| T-2-03-03 | Information Disclosure | Unauthenticated read of invite-only calendar | mitigate | Public read rule requires `visibility == 'public'`; invite-only calendars fail this check; creator read requires auth |
| T-2-03-04 | Tampering | Client direct write to bookings (bypass atomic transaction) | mitigate | `allow write: if false` on bookings; Admin SDK is the only writer; prevents BOOK-05 race conditions from client side |
| T-2-03-05 | Information Disclosure | Client reading OTP tokens from invites collection | mitigate | `allow read, write: if false` on invites; Admin SDK handles all OTP operations (INV-03/INV-04) |
| T-2-03-06 | Elevation of Privilege | Attacker sets own creatorId on created calendar | accept/mitigate | Create rule checks `request.auth.uid == request.resource.data.creatorId`; attacker can only set their own uid (not impersonate another creator); this is the intended behavior |
</threat_model>
