<interfaces>
Current Calendar.tsx theme logic (BUGGY — to refactor):
- `getEffectiveTheme()` called synchronously at render (line 46); guards window inside, returns 'light' on server.
- A `useEffect` (lines 49-56) calls bare `window.matchMedia` with deps `[theme, state]`. `state` is recreated every render → effect re-runs every render → calls `state.setView(state.view)` → re-render loop risk.

Target refactor (from RESEARCH.md Pattern 5):
```typescript
const [resolvedTheme, setResolvedTheme] = useState<'light' | 'dark'>(() =>
  theme === 'auto' ? 'light' : theme   // server default = 'light' → no hydration mismatch
)
useEffect(() => {
  if (theme !== 'auto') { setResolvedTheme(theme); return }
  if (typeof window === 'undefined' || !window.matchMedia) return
  const mq = window.matchMedia('(prefers-color-scheme: dark)')
  setResolvedTheme(mq.matches ? 'dark' : 'light')
  const handler = (e: MediaQueryListEvent) => setResolvedTheme(e.matches ? 'dark' : 'light')
  mq.addEventListener('change', handler)
  return () => mq.removeEventListener('change', handler)
}, [theme])   // NOTE: no `state` in deps — fixes infinite-render bug
```
Then use `resolvedTheme` where `effectiveTheme` was used (rootClass + Provider value `theme: resolvedTheme`).

Demo app current Slot mapping (apps/web/src/App.tsx ~line 24): maps storedSlots to `{ ..., startTime: s.startTime, endTime: s.endTime, ... }`. StoredSlot (appStore.ts) has `startTime`/`endTime` "HH:MM" + a seed function.
