## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps inside buttons/inputs, inline padding |
| sm | 8px | Compact stacking (label → input gap, helper text gap) |
| md | 16px | Default form-field vertical spacing, card internal padding (mobile) |
| lg | 24px | Card internal padding (desktop), spacing between form and footer links |
| xl | 32px | Vertical gap between `AuthLayout`'s logo/heading block and the card |
| 2xl | 48px | Top/bottom page padding on `AuthLayout` at desktop viewport |
| 3xl | 64px | Not used in this phase (no full-page marketing sections) |

Exceptions:
- Icon-only buttons (Google "G" glyph button, password show/hide toggle) use a 32px (`size-8`, shadcn default `icon` button size) square touch target — below the 44px mobile-accessibility guideline but consistent with shadcn's dense default scale used throughout; acceptable for desktop-first auth flows, executor should confirm tap comfort on mobile during manual QA.
- Skeleton loading block (`Guard.tsx` loading state) uses `md` (16px) internal gaps between skeleton lines, not a spacing scale violation, just noting non-content nature.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 |
| Label | 14px (`text-sm`) | 500 (medium) | 1.4 |
| Heading | 20px (`text-xl`) | 600 (semibold) | 1.2 |
| Display | 28px (`text-3xl`... use `text-[1.75rem]`/`text-3xl`) | 600 (semibold) | 1.2 |

Notes:
- Two weights only, per design-contract rule: 400 (regular, body copy, helper/error text) and 600 (semibold, headings, card titles, "Sign in" / "Create account" page titles). shadcn's `button.tsx` default uses `font-medium` (500) for button labels — this is the one sanctioned exception (interactive control labels), not a third body/heading weight; do not introduce 500 elsewhere.
- Display (28px/600) is reserved for the single page-level heading at the top of each `AuthLayout` card (e.g. "Welcome back", "Create your account", "Reset your password"). Heading (20px/600) is reserved for sub-sections (e.g. card title inside `CardHeader` if distinct from the page-level heading — in practice these will usually collapse into one element; do not stack both on the same page).
- Body line-height 1.5 (form helper text, error messages, empty-state copy). Heading/Display line-height 1.2.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `oklch(1 0 0)` / `--background` (white) light mode, `oklch(0.145 0 0)` dark mode | Page background behind `AuthLayout`, body text contrast `--foreground` |
| Secondary (30%) | `oklch(1 0 0)` / `--card` (white, light) — visually distinct from background only via `--border` (`oklch(0.922 0 0)`) and subtle shadow | The centered auth card itself (`Card` component), and `--muted` (`oklch(0.97 0 0)`) for input backgrounds/disabled states |
| Accent (10%) | `--primary` = `oklch(0.205 0 0)` (near-black in light mode) | Reserved EXCLUSIVELY for: primary CTA buttons ("Create account", "Sign in", "Send reset link", "Save password"), active focus ring (`--ring`), and links inside body copy (e.g. "Forgot password?", "Already have an account? Sign in") |
| Destructive | `--destructive` = `oklch(0.577 0.245 27.325)` (red) | Reserved EXCLUSIVELY for: inline form validation errors, the centralized Firebase auth-error `Alert` banner (D-14), and the password-change re-authentication failure state — never used for non-error emphasis |

Accent reserved for: primary submit buttons on each auth form, the focus ring on the currently-focused input, and inline text links connecting auth pages to each other (sign-in ↔ sign-up, forgot-password back-link). Google sign-in button uses `outline` button variant (neutral border, not accent) so it visually reads as a secondary action relative to the primary email/password CTA — this satisfies the "one obvious default action per page" rule.

This is a grayscale-dominant palette by design (shadcn `base-nova` neutral preset, no project brand color decided yet) — 60/30/10 here refers to surface/elevation hierarchy (background vs. card vs. interactive-accent), not hue variety. If a brand color is chosen in a future phase, only the `--primary`/accent role should be re-themed; do not introduce a second accent.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA — Sign up | "Create account" |
| Primary CTA — Sign in | "Sign in" |
| Primary CTA — Forgot password | "Send reset link" |
| Primary CTA — Change password | "Update password" |
| Secondary CTA — Google | "Continue with Google" |
| Empty state heading | Not applicable — auth pages have no data-driven empty states (forms always render); `/account/security` page itself has no "empty" condition |
| Empty state body | N/A (see above) |
| Verify-email pending state heading | "Check your email" |
| Verify-email pending state body | "We sent a verification link to {email}. Click the link to verify your account, then return here." + resend action: "Resend email" (disabled with countdown for 60s after each send, per RESEARCH.md Open Question §3 recommendation) |
| Error state — generic auth failure | Centralized message from `lib/firebase/auth-errors.ts` (D-14), e.g. "Incorrect email or password." shown in an `Alert variant="destructive"` directly above the submit button — never a raw Firebase error string |
| Error state — network failure | "Network error. Please check your connection and try again." (already in RESEARCH.md's error map) |
| Error state — re-auth required (password change) | "For security, please sign in again before changing your password." followed by a re-auth prompt (current-password field) inline on the same page, not a separate route |
| Destructive confirmation — none in this phase | No delete/destroy actions exist in Phase 3 (account deletion is out of scope; password change is a mutation, not destructive-confirm-worthy). N/A. |
| Onboarding username step heading | "Choose a username" |
| Onboarding username step body | "This will be your public profile URL: easyappointment.app/u/{username}" (live preview as user types) |
